# Preview — see a component in isolation, no app, no npm

Read this when you want a Storybook-style catalogue of components without the
Storybook. It's a thin harness over the component factory contract you already
follow (`create<Name>(props, signal) → { el, … }`): a factory that self-loads its
own template + CSS, takes no global state, and imports no view is *already* an
isolatable unit, so previewing it is almost free.

## Glossary

- **preview** — the feature, and the file that declares how to preview one
  component: `<name>.preview.js`, colocated in the component folder.
- **variant** — a named prop-set inside a preview (`default`, `long`, `empty`).
  Each variant renders as one labeled frame. _Distinct from_ CSS/design "variants".

Components only. Views are route-bound and data-backed — faking that is where
lean storybooks rot, so previews stop at the component boundary.

## What a preview file looks like

Colocate `<name>.preview.js` with the component. Default-export
`{ title, render, variants }`:

```js
// components/stat-card/stat-card.preview.js
// @ts-check
import { createStatCard } from "./stat-card.js";

/** @type {import("../../preview.js").PreviewModule} */
export default {
  title: "stat-card",
  // render adapts the factory's return to an element — the harness stays
  // factory-agnostic and this awaits the async ensure() inside the factory.
  render: (props, signal) => createStatCard(props, signal).then((c) => c.el),
  variants: {
    default: { label: "Runs", value: "0" },
    long: { label: "Active background sessions", value: "1,284" },
    zero: { label: "Errors", value: "0" },
  },
};
```

The `@type {… PreviewModule}` annotation is the one bit of ceremony: under the
`tsc` gate it contextually types `render`'s params (no implicit-`any`) and gives
editor autocomplete for the shape. The path is relative to the component folder
(`../../preview.js` from `components/<name>/`).

## Scaffold a new one (don't hand-write the boilerplate)

`previews/new.mjs` seeds that file for you — it fills the mechanical parts (the
imports, the easy-to-get-wrong `@type` path, the `render` adapter, `title`) and
leaves a single `default` variant with a `TODO`:

```sh
node previews/new.mjs stat-card   # → components/stat-card/stat-card.preview.js
```

This is a one-time **seed**, the opposite contract to the generated registry: it
refuses to overwrite an existing file, and once written it's yours to edit. It's a
separate step **in front of** the generator — scaffold, fill in `variants`, then
let the generator catalogue it. The factory name is generated by convention
(`stat-card` → `createStatCard`); if your names differ, the `tsc` gate flags it
and you fix the seed (the scaffolder stays dumb — it never parses your source).

`render(props, signal)` returns an `Element` or a `Promise<Element>`. The `signal`
is the same teardown contract the app shell hands views: any listener the
component attaches with it dies when you switch to another component. Props are
shown verbatim under each frame as JSON, so the catalogue doubles as docs
(callbacks/functions simply don't appear in that caption).

## How discovery works (and why it's codegen)

The browser has no reflection — native ES modules with no build step can't
enumerate files; a module only exists once something `import`s its exact URL. So
the import list has to be emitted by something with filesystem access. The three
options and why this one:

- **build-time glob** (Vite `import.meta.glob`, Storybook) — needs a bundler,
  which this stack doesn't have. Out.
- **runtime server walk** — works, but the list is invisible to `tsc` and not
  greppable.
- **codegen** (chosen) — `previews/scan.mjs` walks `components/` for
  `*.preview.js` and writes `previews/registry.js`. The output is real source:
  `tsc` checks it, `grep` finds it, and it's committed. This is the no-build
  equivalent of a **C# source generator**.

`previews/registry.js` is generated — never hand-edit it:

```js
// GENERATED by previews/scan.mjs — do not edit by hand.
export const previews = [
  { title: "stat-card", load: () => import("../components/stat-card/stat-card.preview.js") },
];
```

`title` is the component folder name (pure path work — `scan.mjs` never executes
a preview module, so a broken preview can't break discovery). The file changes
only when the *set* of previews changes, so there's no git churn on ordinary
edits — and when it does change, commit it alongside the component.

## Running it

Two steps with deliberately opposite contracts: **scaffold** (one-time seed) then
**generate** (continuous). Generation is on by default and driven by the dev server:

```sh
node previews/new.mjs <name>   # 1. seed a *.preview.js (separate, one-time)
node serve.mjs                 # 2. regenerates the registry, then serves
                               #      open http://127.0.0.1:8080/preview.html
PREVIEW=off node serve.mjs     # skip generation (e.g. prod static serving)
node previews/scan.mjs         # manual fallback — regenerate without serving
```

Add a `*.preview.js`, restart the dev server, and it appears in the rail. The
preview page is standalone (`preview.html`) — it is not wired into the app router,
so nothing preview-related ships in the real bundle.

## The page

- Left **rail** lists components (alphabetical). Selecting one sets
  `location.hash = "#/<title>"` (deep-linkable, back button works).
- The **canvas** renders *every* variant of the selected component stacked, each
  in a labeled frame with its props caption — so you see the whole matrix at once.
- **Theme toggle** (auto / light / dark) flips `color-scheme`, so `light-dark()`
  tokens are exercised both ways — same idiom as `shell.js`.
- A variant that throws shows its error in-frame instead of blanking the canvas.

## Files (copy verbatim, like the other canonical pieces)

- `preview.html` — the standalone page + harness `<template>`s
- `preview.js` — the harness (rail, canvas, theme, teardown)
- `preview.css` — chrome, `light-dark()`-themed, self-sufficient
- `previews/new.mjs` — the scaffolder (one-time seed of a `*.preview.js`)
- `previews/scan.mjs` — the generator (run by `serve.mjs`, or by hand)

## Deliberately out of scope (keep it lean)

No addons, no auto-generated controls/args editor, no MDX docs, no test-runner
hookup, no build step. Resizable `@container` frames and a background swatch were
considered and deferred — add them only if eyeballing demands it.
