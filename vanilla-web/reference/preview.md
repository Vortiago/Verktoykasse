# Preview — see a component in isolation, no app, no npm

Read this when you want a Storybook-style catalogue of components without the
Storybook.

## Glossary

- **preview** — the feature, and the file that declares how to preview one
  component: `<name>.preview.js`, colocated in the component folder.
- **variant** — a named prop-set inside a preview (`default`, `long`, `empty`).
  Each variant renders as one labeled frame. _Distinct from_ CSS/design "variants".

Components only. Views are route-bound and data-backed.

## What a preview file looks like

Colocate `<name>.preview.js` with the component. Default-export
`{ title, render, variants }`:

```js
// components/stat-card/stat-card.preview.js
// @ts-check
import { createStatCard } from "./stat-card.js";

/** @type {import("../../preview.js").PreviewModule} */
export default {
  title: "stat-card",
  // render adapts the factory's return to an element — the harness stays
  // factory-agnostic and this awaits the async ensure() inside the factory.
  render: (props, signal) => createStatCard(props, signal).then((c) => c.el),
  variants: {
    default: { label: "Runs", value: "0" },
    long: { label: "Active background sessions", value: "1,284" },
    zero: { label: "Errors", value: "0" },
  },
};
```

The `@type {… PreviewModule}` annotation is the one bit of ceremony: under the
`tsc` gate it contextually types `render`'s params (no implicit-`any`) and gives
editor autocomplete for the shape. The path is relative to the component folder
(`../../preview.js` from `components/<name>/`).

## Scaffold a new one (don't hand-write the boilerplate)

`previews/new.mjs` seeds that file for you — it fills the mechanical parts (the
imports, the easy-to-get-wrong `@type` path, the `render` adapter, `title`) and
leaves a single `default` variant with a `TODO`:

```sh
node previews/new.mjs stat-card   # → components/stat-card/stat-card.preview.js
```

This is a one-time **seed**, the opposite contract to the generated registry: it
refuses to overwrite an existing file, and once written it's yours to edit. It's a
separate step **in front of** the generator — scaffold, fill in `variants`, then
let the generator catalogue it. The factory name is generated by convention
(`stat-card` → `createStatCard`, via `previews/naming.mjs` — the one place this
runs, shared with the usage snippet below so the two can't drift apart); if
your names differ, the `tsc` gate flags it and you fix the seed (the
scaffolder stays dumb — it never parses your source).

`render(props, signal)` returns an `Element` or a `Promise<Element>`. The `signal`
is the same teardown contract the app shell hands views: any listener the
component attaches with it dies when you switch to another component. Each
frame's caption shows the actual call you'd write for that variant — e.g.
`createStatCard({ label: "Runs", value: "0" })` — reconstructed from the title
and that variant's props (see **Usage snippet**, below), so the catalogue
doubles as docs without any extra authoring.

## How discovery works (and why it's codegen)

The browser has no reflection — native ES modules with no build step can't
enumerate files; a module only exists once something `import`s its exact URL. So
the import list has to be emitted by something with filesystem access:
`previews/scan.mjs` walks `components/` for `*.preview.js` and writes
`previews/registry.js` — real source that `tsc` checks and `grep` finds.

`previews/registry.js` is generated — never hand-edit it:

```js
// GENERATED by previews/scan.mjs — do not edit by hand.
export const previews = [
  { title: "stat-card", dir: "components/stat-card",
    load: () => import("../components/stat-card/stat-card.preview.js") },
];
```

`title` is the component folder name (pure path work — `scan.mjs` never executes
a preview module, so a broken preview can't break discovery). `dir` is that
folder's path relative to the web root — the harness `fetch()`s
`${dir}/${title}.{html,css,js}` from it (see **View source**, below). The file
changes only when the *set* of previews changes — commit it alongside the component.

## Running it

Two steps with deliberately opposite contracts: **scaffold** (one-time seed) then
**generate** (continuous). Generation is on by default and driven by the dev server:

```sh
node previews/new.mjs <name>   # 1. seed a *.preview.js (separate, one-time)
node serve.mjs                 # 2. regenerates the registry, then serves
                               #      open http://127.0.0.1:8080/preview.html
PREVIEW=off node serve.mjs     # skip generation (e.g. prod static serving)
node previews/scan.mjs         # manual fallback — regenerate without serving
```

Add a `*.preview.js`, restart the dev server, and it appears in the rail. The
preview page is standalone (`preview.html`) — it is not wired into the app router,
so nothing preview-related ships in the real bundle.

## The page

- Left **rail** lists components (alphabetical). Selecting one sets
  `location.hash = "#/<title>"` (deep-linkable, back button works).
- The **canvas** renders *every* variant of the selected component stacked, each
  in a labeled frame with its usage-snippet caption — so you see the whole
  matrix at once.
- **Theme toggle** (auto / light / dark) flips `color-scheme`, so `light-dark()`
  tokens are exercised both ways — same idiom as `shell.js`.
- A variant that throws shows its error in-frame instead of blanking the canvas.

## Usage snippet

Each frame's caption shows the call you'd actually write for that variant —
e.g. `createStatCard({ label: "Runs", value: "0" })` — instead of raw JSON
props. It's reconstructed, not authored: the argument is that variant's
`props` formatted as an idiomatic object literal (unquoted keys,
callback/function props omitted — there's nothing meaningful to show for
one), and since it's read straight from the real variant, no per-component
authoring, it can't drift from it. The factory name comes from `title` via
`previews/naming.mjs`'s `factoryNameFor` — the same function
`previews/new.mjs` uses to scaffold the real import (see **Scaffold a new
one**, above), so those two callers can't drift **from each other**. That's
narrower than it sounds: nothing checks the printed name against what a
component's own module actually exports, so a factory hand-renamed off the
`create<Name>` convention still gets a confidently wrong "copy this" snippet.

This is the same idea as Storybook's "Show code": a snippet attached to each
individual story, not one shared example — since usage genuinely differs per
variant (unlike the files under **View source**, which are identical across
variants).

## View source

The widget behind this tab strip lives in its own module, `preview-source.js`
(`wireSourceTabs`, imported and mounted once per shown component by
`preview.js`) — everything below is that module's behavior.

A `Preview | HTML | CSS | JS` tab strip sits above the frame stack — **once per
component**, not per variant, since the underlying files are identical across
every variant of a component. `HTML`/`CSS`/`JS` swap the frame stack out for
that file's real, verbatim content (`fetch()`ed via the registry's `dir` field,
lazily, cached per component) — what you see is exactly what you'd copy into
your own app, same as the rest of this copy-verbatim library. Unlike the
per-variant caption (see **Usage snippet**, above), nothing here is
reconstructed — the files themselves are the catalogue.

Highlighting is intentionally minimal — comments and strings only, no
keywords/tags/properties — via the native **CSS Custom Highlight API**
(`CSS.highlights` / `::highlight()`, Chrome/Edge since version 105) rather than
an external tokenizer/highlighter library. A hand-rolled regex per language
finds comment and string ranges in the fetched text and registers them as two
`Highlight` objects; the CSS lives in `preview.css` as
`::highlight(src-comment)` / `::highlight(src-string)`. A copy button next to
the tabs puts the active tab's raw text on the clipboard.

**Known limitation** (accepted, not a bug to chase): the string-detection regex
has no concept of a JS regex literal or a template-literal `${…}` interpolation.
A quote inside either can make the tokenizer misread real code that follows as
still being inside a string, or swallow an interpolated expression whole as
one flat string run. No component in this catalogue triggers it today, but if
one adds a regex/template literal with an embedded quote, its JS tab may
mis-highlight — cosmetic only, never a crash.

## Files (copy verbatim, like the other canonical pieces)

- `preview.html` — the standalone page + harness `<template>`s
- `preview.js` — the harness (rail, canvas, usage snippet, theme, teardown)
- `preview-source.js` — the view-source widget (fetch, tokenize, CSS Custom
  Highlight, clipboard) behind `preview.js`'s tab strip
- `preview.css` — chrome, `light-dark()`-themed, self-sufficient
- `previews/new.mjs` — the scaffolder (one-time seed of a `*.preview.js`)
- `previews/scan.mjs` — the generator (run by `serve.mjs`, or by hand)
- `previews/naming.mjs` — the shared `factoryNameFor` convention (new.mjs + preview.js)

## Deliberately out of scope (keep it lean)

No auto-generated controls/args editor, no MDX docs, no test-runner hookup, no
build step, no external syntax-highlighting/tokenizer library (see **View
source**, above — full keyword/tag/property highlighting was deliberately
dropped in favor of a minimal hand-rolled comment/string pass). Resizable
`@container` frames and a background swatch were considered and deferred — add
them only if eyeballing demands it.
