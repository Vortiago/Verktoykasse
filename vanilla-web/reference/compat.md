# Compat — per-feature browser support & degradation

Read this before shipping an app for a stated audience wider than "my own
tools." Both skills lean on modern platform features; this page is the one
place to check what degrades gracefully vs. what silently breaks, instead of
re-deriving it from scattered inline asides (`tooltip: Chromium 125+`,
`moveBefore (Chromium 133+)`, …) each of which drifts independently. **This
page is the source; treat any version number still inlined elsewhere in
SKILL.md/`reference/*.md` as advisory and this table as current.**

## Stance

- **Primary target is Chrome/Edge** — the maintainer's own tools, used by
  their author. Nothing here gates adoption of a feature for that audience.
- This matrix exists for the day an app declares a wider audience
  ("must work in Firefox/Safari", a public-facing tool, an embed). Read the
  **Without it** column for that call: `graceful` (ship as-is, the fallback is
  load-bearing and already in the code), `degraded` (ships, loses polish —
  a judgment call per app), `broken` (do not ship to that audience without
  swapping the feature out).
- Entries are dated **as of 2026-07**, not chased to exact versions — browsers
  keep shipping. Support is Baseline status or coarse browser+version,
  verified against MDN/BCD at authoring time (noted per row); re-check BCD
  before trusting an old row for a launch decision.
- A lot of this list shipped cross-browser in the last ~12 months (see below);
  the "Chromium-only" framing that motivated this page (#43) is now stale for
  several rows. The gaps that are *still* live as of 2026-07 are `moveBefore`,
  `dialog closedby`, `popover="hint"`, cross-document `@view-transition`, and
  the newest CSS (`scroll-marker-group`, `appearance: base-select`,
  `animation-timeline`) — those are this doc's real spine.

## The matrix

| Feature (where used) | Support (as of 2026-07) | Without it | Guidance |
|---|---|---|---|
| **CSS anchor positioning** — `anchor-name`/`position-anchor`/`position-area` (`tooltip`, `menu` components) | Chrome/Edge 125 (May 2024), **Firefox 147 (Jan 2026), Safari 26 (Sep 2025)** — newly cross-browser as of this window | **broken** on an engine without it: the tethered element renders at the UA's fallback position (browser-dependent, often top-left or `static`), not "unstyled but usable" | For Chrome/Edge-only apps, use as-is. For an audience that must cover pre-2026 Safari/Firefox, position on the popover's `beforetoggle` event in JS instead (`anchorPopover`, `reference/modules.md`) — this is *why* that helper exists, kept as the fallback path even though native support has since landed broadly |
| **`moveBefore()`** — `reconcileList`'s state-preserving row move (`lib/templates.js`) | Chrome/Edge 133 (Feb 2025), Firefox 144 (Oct 2025). **Not in Safari** (WebKit bug open, unshipped) | **graceful** — `reconcileList` feature-detects and falls back to `insertBefore` (`lib/templates.js`): the row still moves, just loses focus/selection/scroll/running-animation preservation across the move | Safari-inclusive audience: fine to ship, just don't promise state-preserving reorder as a UX guarantee on Safari yet. Chrome/Edge/Firefox users get the full behavior today |
| **`@scope`** — every component/view stylesheet | Baseline widely available: Chrome/Edge 118 (Oct 2023), Safari 17.4 (Mar 2024), Firefox 146 (Dec 2025) | n/a — no fallback path exists or is needed; this is load-bearing to the whole CSS convention | Ship anywhere. Firefox was the last holdout and caught up Dec 2025; nothing here gates on browser anymore |
| **`popover` attribute** (`auto`/`manual`) — `menu`, `tooltip` overlay base | Baseline: Chrome/Edge 114 (May 2023), Safari 17 (Sep 2023), Firefox 125 (Apr 2024) | n/a | Ship anywhere |
| **`popover="hint"`** — `tooltip`'s top-layer stacking against an open `auto` popover (adopted, feature-detected) | Chrome/Edge 133 + Firefox 149 (Mar 2026) both implement an **older, inconsistent** draft of the spec (their own release notes flag it); **not in Safari at all** | **graceful**: `tooltip.js` sets `el.popover = "hint"` then reads it back (the `popover` IDL is limited-to-known-values, so an unsupported value simply doesn't stick) and falls back to `"manual"` when it didn't — an engine without `hint` gets the same hand-rolled show/hide behavior the component shipped with before this adoption | Adopted for the Chrome/Edge-primary audience, with the fallback covering everyone else. The cross-browser caveat still holds for a wider audience: Chrome/Edge and Firefox implement different draft readings of the spec, so don't rely on `hint`'s exact stacking semantics matching across those two — only that the fallback keeps every engine usable |
| **`light-dark()`** — every color token in `tokens.css`/`shell.css` | Baseline: Chrome/Edge 123 (Mar 2024), Safari 17.5 (May 2024), Firefox 120 (Nov 2023) | n/a | Ship anywhere |
| **`<dialog>` `closedby`** — `createDialog`'s `closeOnBackdrop` (`closedby="any"`) | Chrome/Edge 134 (Mar 2025), Firefox 141 (Jul 2025). **Safari: preview only, not in a stable release** | **degraded**: without it the dialog just doesn't light-dismiss on backdrop click — ESC and the explicit close button still work (native `<dialog>` gives those for free regardless) | Safari-inclusive audience: don't rely on backdrop-click dismiss as the only close affordance — the dialog already ships a ✕ button via Invoker Commands, so nothing is actually stranded; it's a convenience gap, not a broken dialog |
| **Invoker Commands** — `command`/`commandfor` incl. custom `--` commands, `CommandEvent` (`dialog`'s ✕/actions, declarative overlay triggers, delegated app actions) | Chrome/Edge 135 (Apr 2025), Firefox 144 (Oct 2025), **Safari 26.2 (Dec 2025)** — newly cross-browser | **broken** on an engine without it: the button fires no `CommandEvent` and invokes nothing — there is no automatic fallback (custom `--` commands have no non-JS equivalent by design) | Was Chrome/Edge-only when `reference/interactivity.md` was written; as of 2026-07 all three engines ship it. For an audience that must cover pre-window browsers, wire a plain `onclick` alongside for the custom-command case; the built-in dialog `show-modal`/`close`/`request-close` values are now safe to rely on across all three |
| **View Transitions — same-document** — `withTransition` (`templates.js`), `document.startViewTransition` | Baseline: Chrome/Edge 111 (Mar 2023), Safari 18 (Sep 2024), Firefox 144 (Oct 2025) | **graceful by construction**: `withTransition` returns a resolved-`finished` shim where the API is missing, so the DOM update still runs — it just isn't animated | Ship anywhere; the helper was written fallback-first, so there's no audience this blocks |
| **View Transitions — cross-document** — `@view-transition { navigation: auto; }` (MPA navigations, not the SPA shell) | Chrome/Edge 126 (Jun 2024), Safari 18.2 (Dec 2024). **Not in Firefox** (unimplemented) | **graceful**: an engine that doesn't know the at-rule just navigates without the crossfade — a normal page load, nothing broken | Only relevant to a tool that navigates between real documents (the canonical `shell.js` is an SPA and never hits this path). Safe to declare unconditionally |
| **`AbortSignal.any()`** — `api-client.js`'s `request()` composes a caller signal (a view's mount signal) with the default timeout signal, so either can abort (#48) | Baseline, and has been for a while: Chrome/Edge 116 (Aug 2023), Safari 17.4 (Mar 2024), Firefox 124 (Mar 2024); Node 20.3+ | **broken** without it: no feature-detect or fallback — an engine (or a Node runtime) older than the versions at left throws, same failure mode as any other un-polyfilled static method in this codebase | Adopted directly, no feature-detect — it's been Baseline across all three engines (and the Node versions this repo targets) long enough that gating it would be pure ceremony |
| **`Intl.DurationFormat`** — `format.js`'s `duration()`, now locale-aware via `{ style: "narrow" }` with `secondsDisplay`/`minutesDisplay: "always"` to keep zero-valued units visible (#49) | Baseline: Chrome/Edge 129 (Sep 2024), Firefox 136 (Mar 2025), **Safari shipped first, 16.4 (Mar 2023)**; Node 23+ | **broken** without it: no feature-detect or fallback — an engine (or a Node runtime) that lacks the constructor throws, same as any other un-polyfilled `Intl.*` call in this codebase | Adopted because it's Baseline across all three engines; the only live constraint is the Node version running tests/CI (Node 23+, or `--harmony-intl-duration-format` on an older 20/22) — see `.github/workflows/gate.yml`'s `node-version` for what's actually pinned; it's a non-issue as long as that stays ≥23. A locally-typed shim (`DurationFormatLike`) papers over lib.d.ts lagging Baseline, it doesn't change runtime behavior |
| **Navigation API** — not adopted; `shell.js` routes on `hashchange`. See #51 (explore, not committed) | Chrome/Edge 102 (2022), **Firefox 147 (Jan 2026), Safari 26.2 (Dec 2025)** — newly cross-browser | n/a (not adopted) | #51's own tradeoff still holds even though the compat gap that motivated "explore, not do" has closed: path-based routing needs a static-host rewrite rule that hash routing doesn't, and that's a deployment-surface change (see `reference/deploy.md`), not just a compat one. Cross-browser support removes one blocker, not the other |
| **Trusted Types** — `require-trusted-types-for 'script'` + the `vanilla-templates` policy (see `reference/security.md`) enforced by `serve.mjs`'s CSP | Chrome/Edge 83 (2020) long-standing; **Firefox 148 (Feb 2026), Safari 26 (Sep 2025)** — now shipped in all three as of this window | **broken without a policy**: an engine that enforces the header and hits an unpoliced `innerHTML`/`insertAdjacentHTML` throws a `TypeError` at that call, not a silent no-op | `loadTemplates` owns the single named policy so this is a non-issue for app code that stays inside the conventions; only a hand-rolled trusted-HTML sink outside `lib/templates.js` needs a look, see `reference/security.md`'s loosening note |
| **`appearance: base-select`** — not currently used by any component; noted here as the native alternative to a hand-rolled combobox (`reference/interactivity.md`) | Chrome/Edge 135 (Apr 2025); **Firefox only behind two flags (149, Mar 2026)**; **Safari 27 (unreleased at authoring time)** | **graceful**: an unsupported engine renders the plain native `<select>` — still fully usable, just unstyled | Fine to reach for on a Chrome/Edge-only app today; a wider audience gets a plain-but-working select as the fallback, which is an acceptable default even before adopting the styled version |
| **`scroll-marker-group` / `::scroll-marker` / `::scroll-button()`** — carousel dots + prev/next (`reference/css.md`) | Chrome/Edge 135 (Apr 2025). **Not in Firefox or Safari** | **graceful**: unsupported engines fall back to a plain `scroll-snap` strip — still swipeable, just no dot/arrow chrome | Safe as a progressive enhancement on any audience; don't build a JS carousel just to cover the gap |
| **`animation-timeline` (scroll-driven animations)** — reading-progress bar, reveal-on-scroll (`reference/css.md`) | Chrome/Edge 115 (Jul 2023), Safari 26 (Sep 2025). **Firefox: behind a flag only, no shipped version** | **graceful**: the resting (non-animated) state stays usable — the recipe explicitly says "keep the resting state usable without it" | Safe as a progressive enhancement; don't gate a feature's usability on the animation running |
| **`content-visibility: auto`** — the native virtual-list recipe (`reference/css.md`) | Baseline: Chrome/Edge 85 (2020), Firefox 125 (Apr 2024), Safari 18 (Sep 2024, with a find-in-page caveat fixed in Safari 26) | **graceful**: an engine without it just renders every row — correct, only loses the off-screen skip that keeps a huge list cheap | Ship anywhere; on a pre-2024-Safari/Firefox audience a very large list (thousands of rows) pays full render cost, which is a performance-only degradation, not a correctness one |
| **`URLPattern`** — the suggested param-route matcher if path routing is ever adopted (#51) | Chrome/Edge 95 (2021), **Firefox 142 (Aug 2025), Safari 26 (Sep 2025)** — cross-browser as of this window | n/a (not adopted; only relevant if #51 lands) | Removes the "don't hand-roll regex" concern #51 raised — no polyfill needed for any current target if that issue is picked up |
| **CSS Custom Highlight API** — `preview.html`'s source-view syntax highlighting (`CSS.highlights`/`::highlight()`) | Baseline: Chrome/Edge 105 (2022), Safari 17.2 (Dec 2023), Firefox 149 (Mar 2026, partial from 140) | **degraded** where unsupported/partial: an older engine just shows unhighlighted source text, or (pre-149 Firefox) highlights that don't combine with `text-shadow`/`text-decoration` — the copy-to-clipboard and raw content are unaffected either way | Preview catalogue is a local dev tool, not shipped app UI; the gap is now closed anyway as of 2026-07 |
