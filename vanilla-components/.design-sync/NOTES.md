# design-sync notes — vanilla-components

This is an **off-envelope** sync: vanilla-components is a no-React vanilla-web
library. design-sync is React-only, so we feed the converter a **React-shim
adapter** generated by `bridge/emit-adapter.mjs` into `bridge/ds-adapter/`.

## How to re-sync

1. `node bridge/emit-adapter.mjs` — regenerates the adapter from the real
   component sources (factory verbatim; only the dev-server self-loaders are
   neutralized — template inlined + registered lazily, `loadCSS` → no-op,
   component CSS shipped via `styles.css`).
2. Ensure adapter deps: `(cd bridge/ds-adapter && npm i react react-dom -D @types/react)`
   (the generator wipes `dist/`+`css/` only, not `node_modules`).
3. Build/validate from `vanilla-components/`:
   ```
   node .ds-sync/package-build.mjs --config .design-sync/config.json \
     --node-modules bridge/ds-adapter/node_modules --entry bridge/ds-adapter/dist/index.js --out ./ds-bundle
   node .ds-sync/package-validate.mjs ./ds-bundle
   ```
- `cssEntry` is **package-relative to the adapter** = `styles.css` (a *compiled*
  stylesheet — tokens + every component's `@scope` block inlined; NOT `@import`
  stubs, which the converter copies verbatim without following).
- Render-check: playwright **1.59.1** matches the cached chromium-1217.

## Known render warns (triaged benign)

- `[FONT_MISSING] "Cascadia Code"` — the `--mono` token lists Cascadia Code as a
  preference; no `@font-face` ships it. **Accepted**: it's a dev-preference mono,
  not a brand font, and the stack falls back to `ui-monospace`/system mono (the
  stat-card numbers render fine in the screenshots). Not a brand-fidelity issue.

## Component mapping

- `Panel`, `StatCard`, `Chip`, `StatusDot`, `AppBar`, `SideNav`, `ViewHeader` are
  declarative — clean React shims (the default `declarativeShim`).
- `Tooltip` is **imperative** (a hover overlay). Its shim mounts a host and shows
  the bubble on mount so the card isn't blank — it's a demo of the primitive, not
  a declarative `<Tooltip>` API. A future interactive wrapper (trigger + hover)
  could replace it.
- **Inter-component imports**: `side-nav` imports `createChip` from `../chip/chip.js`.
  The adapter dist is flat, so `emit-adapter.mjs`'s `neutralize()` rewrites
  `../<name>/<name>.js` → `./<name>.js`. Any new cross-component import relies on
  that rewrite — keep the `<dir>/<file>` names matching (the regex assumes it).

## Re-sync risks

- The adapter (`bridge/ds-adapter/`) is **generated** and gitignored — it must be
  regenerated (step 1) before any re-sync; the durable inputs are the real
  components + `.design-sync/` (config, previews, conventions.md, this file).
- Adapter `node_modules` (react/react-dom/@types/react) and `.ds-sync/` deps are
  installed from npm — a fresh clone needs the installs re-run.
- The shim pins React via `import "react"`; the converter externalizes it to
  `_vendor/` so the design runtime's React is used. If a future React API breaks
  `useEffect`/`useRef` semantics, the shim is where to look.
